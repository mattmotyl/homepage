3. What do platforms generate from collected data?

The event logs described in Section 2 power everything from recommendation systems to integrity defenses. Platforms use the raw signals to predict which videos will be watched, which accounts are trustworthy, and whether a piece of content likely violates policy. This page retains the full narrative from the report while adding plain-language explanations so you can follow each concept without needing a background in data engineering.

Storage and processing at VLOPSE scale is expensive. To make queries efficient, teams break activity into specialized tables. Most follow a dimensional model: fact tables capture actions or events, dimension tables describe the people or objects involved, and probabilistic scores summarize model outputs. You do not need to build a warehouse yourself—you simply need to recognize these shapes so you can request the right extracts and interpret the columns you receive.

3.1 Fact tables

Fact tables store discrete events. They are often long and narrow: each row represents a specific action, and the columns capture identifiers that let you connect the event to additional context. The EDMO report uses the following user activity table to illustrate the pattern.

user_id session_id event_id post_id event_time
12 5021 LIKE 90 2025-01-01T00:10:10Z
12 5021 VIEW 93 2025-01-01T00:20:15Z
12 5022 LIKE 93 2025-01-01T00:20:20Z
53 5023 REPLY 143 2025-01-01T00:30:23Z
53 5023 SHARE 143 2025-01-01T00:30:24Z

Even in five rows you can see core analytic moves: two people generated the activity above, one had two sessions, and each session included multiple actions. Fact tables almost always include timestamps, making it possible to group by day or hour to understand trends.

Platforms maintain dictionaries so analysts know what each column means. When Article 40 requests reference documentation, you may receive a table similar to the one below.

Variable Description Data type Retention (days) Restricted?
user_id Assigned numeric ID for each account. INTEGER 730 No
session_id Identifier for a contiguous period of activity. INTEGER 730 No
event_id Label describing the action (view, like, share, etc.). STRING 730 No
post_id Content identifier associated with the event. INTEGER 730 No
event_time Timestamp when the event occurred (UTC). TIMESTAMP 730 No

Note the emphasis on retention windows. Many fact tables only store two years of history for privacy or cost reasons. When designing your research, plan for rolling exports or snapshots if you need longer horizons.

3.2 Dimension tables

Dimension tables enrich the identifiers found in fact tables. A user_id becomes a person with creation date, declared location, and safety flags; a post_id becomes an object with language, media type, and enforcement status. These tables are wide, with dozens or hundreds of columns that describe relatively stable attributes.

User dimension user_id, name, account_age_days, declared_country, risk_tier, consent_status
Post dimension post_id, media_type, caption_language, safety_label, distribution_status, author_id
Session dimension session_id, entry_surface, device_type, network_quality, experiment_bucket

Once you have both fact and dimension tables, you can answer richer questions. For example, you can calculate the number of views per session (fact_user_activity) and then slice by country or risk tier using dim_users.

3.3 Joining data across tables

The relationship between fact and dimension tables is the backbone of platform analytics. You join on shared keys: a user_id in the fact table matches the same user_id in the dimension table, linking events to their context. The queries below are drawn directly from the report and show how to derive insight from the sample data. 

-- Daily active users
SELECT
  CAST(event_time AS DATE) AS event_date,
  COUNT(DISTINCT user_id) AS total_users
FROM fact_user_activity
GROUP BY event_date
ORDER BY event_date;
-- Average number of posts viewed per session
WITH per_session AS (
  SELECT
    session_id,
    COUNT(DISTINCT post_id) AS posts_per_session
  FROM fact_user_activity
  GROUP BY session_id
)
SELECT AVG(posts_per_session) AS avg_posts_per_session
FROM per_session;

When you join to dimensions, always verify that the keys are unique. If the join multiplies your rows unexpectedly, check for duplicate records or missing filters (for example, limiting to active posts only). The goal is not to mirror a platform’s entire warehouse but to extract the slices needed to answer your systemic risk questions.

3.4 Probabilistic variables and model outputs

Platforms increasingly rely on machine learning predictions. These scores are probabilistic: they estimate the likelihood that an account is authentic, a post violates policy, or a video is relevant. Treat them as model outputs, not ground truth. The report highlights two case studies—a location authenticity score and a policy enforcement model—to show how reviewers interpret these signals alongside human labels.

When requesting probabilistic fields, ask for accompanying metadata: the model version, the thresholds that trigger actions, the timestamp of the last human review, and any documented accuracy metrics. These details let you judge whether a score is reliable enough for your analysis or if you should triangulate with observed behavior.

You never need to ask a platform for “lookup tables.” If a field references a code (for example, risk_tier values of 1–5), request the documented definitions as part of your Article 40 submission. Platforms maintain this metadata because their own analysts rely on it.
